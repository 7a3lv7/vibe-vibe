# 5.7 UI 最佳实践

> **本节目标**：掌握如何制定 UI 设计视觉交互规则并约束 AI 遵守，学会为组件库新增和优化组件，理解设计与代码的交互验证工作流。

---

## 小明的新发现

小明用 Skills 管住了 AI 的"设计记忆"（5.6 节），但他很快遇到了新问题。

项目越做越大，shadcn/ui 自带的组件不够用了——比如项目需要一个"统计卡片"组件，显示数字、趋势箭头和百分比变化。shadcn/ui 没有这个组件，AI 每次都从头写，写出来的样式五花八门。

更让他头疼的是，他心里有一套视觉规则：主色不能乱用，间距要统一，交互动效要一致。但这些规则只存在他脑子里，AI 根本不知道。

老师傅看出了问题："你需要的不只是让 AI 记住组件列表，而是建立一套**完整的视觉交互规则**，让 AI 像遵守法律一样遵守它们。"

---

## 第一步：制定视觉交互规则

### 什么是视觉交互规则

视觉交互规则比设计系统（5.6 节讲的颜色、间距、圆角）更进一步——它不仅定义"长什么样"，还定义"怎么动"和"怎么交互"。

老师傅举了个例子：

> 设计系统说：按钮用蓝色，圆角 8px。
> 视觉交互规则说：按钮悬停时颜色加深 10%，点击时缩放 0.98，加载时显示旋转图标且禁用点击，完成后用绿色勾号反馈 1.5 秒后恢复。

看出区别了吗？一个是静态的"样子"，一个是动态的"行为"。

### 怎么制定

老师傅建议把规则分成五个维度：

**1. 颜色使用规则**

```markdown
## 颜色使用规则
- 主操作按钮：只用 primary 色（如"提交""保存"）
- 危险操作按钮：只用 destructive 色（如"删除""重置"）
- 辅助操作按钮：用 secondary 色或 outline 变体（如"取消""返回"）
- 禁止在同一行出现两个 primary 按钮
- 状态色不可混用：success 只用于成功反馈，warning 只用于警告，不要用 warning 色做装饰
```

**2. 间距与布局规则**

```markdown
## 间距与布局规则
- 页面外边距：px-6（移动端 px-4）
- 卡片内边距：p-6
- 同级元素间距：space-y-4（垂直）或 gap-4（水平）
- 表单字段间距：space-y-6
- 分组标题与内容间距：space-y-2
- 页面标题与内容间距：mb-8
```

**3. 交互反馈规则**

```markdown
## 交互反馈规则
- 所有可点击元素必须有 hover 状态变化
- 按钮点击后 200ms 内必须有视觉反馈（颜色变化、缩放或加载动画）
- 异步操作必须显示 loading 状态，且操作期间禁用按钮
- 成功操作用 toast 提示，显示 3 秒后自动消失
- 失败操作用 toast 提示（destructive 变体），需手动关闭
- 破坏性操作（删除、重置）必须二次确认
- 表单提交成功后，清空表单或跳转页面，不要停留在原地
```

**4. 动效规则**

```markdown
## 动效规则
- 页面切换：使用 fade 过渡，时长 200ms
- 弹窗打开/关闭：使用 scale + fade，时长 150ms
- 列表项增删：使用 slide + fade，时长 300ms
- hover 效果：时长不超过 150ms
- 禁止使用超过 500ms 的纯装饰动画
- 所有动效必须支持 prefers-reduced-motion 媒体查询
```

**5. 组件使用规则**

```markdown
## 组件使用规则
- 表单必须用 Form + FormField + FormItem 组合，不可直接写 <input>
- 数据表格必须用 DataTable 组件，支持排序、分页
- 确认弹窗用 AlertDialog，不用 Dialog
- 下拉选择超过 5 个选项时，必须支持搜索
- 图片必须设置 alt 属性和加载失败的 fallback
- 图标只用 Lucide React，不可使用其他图标库
```

小明问："这些规则写好之后放哪里？"

老师傅说："放到你的项目 Skill 里。还记得 5.6 节创建的 `DESIGN_SYSTEM.md` 吗？把这些规则加进去，或者创建一个单独的 `INTERACTION_RULES.md` 放到 `references/` 目录。"

```
.claude/skills/my-project-design/
├── SKILL.md
└── references/
    ├── DESIGN_SYSTEM.md       # 颜色、间距、圆角等静态规范
    └── INTERACTION_RULES.md   # 交互反馈、动效、组件使用规则
```

## 第二步：为组件库新增组件

小明的项目需要一个 `StatsCard` 组件，但 shadcn/ui 没有。老师傅教他三步搞定。

### 2.1 先定义，后实现

不要直接说"帮我写一个统计卡片组件"——这样 AI 自由发挥的空间太大。先把组件的"规格"定义清楚。

小明对 AI 说：

```
请为我创建一个 StatsCard 组件，要求如下：

## 组件规格
- 文件位置：src/components/ui/stats-card.tsx
- 基于 shadcn/ui 的 Card 组件封装
- 使用项目的 Tailwind 设计 tokens（参考 DESIGN_SYSTEM.md）

## Props 定义
- title: string - 统计项名称（如"总用户数"）
- value: string | number - 统计数值
- change?: number - 变化百分比（正数显示绿色向上箭头，负数显示红色向下箭头）
- icon?: LucideIcon - 左侧图标
- loading?: boolean - 加载状态，显示骨架屏

## 交互规则
- hover 时卡片轻微上移（translateY -2px）并加深阴影
- loading 状态使用 Skeleton 组件占位
- 变化百分比为 0 时显示灰色横线图标

## 无障碍
- 使用语义化标签
- 数值变化要有 aria-label 说明趋势
```

### 2.2 遵循组件库的代码风格

老师傅强调，新组件要和 shadcn/ui 现有组件保持一致的代码风格。他让小明在提示词里加上：

```
请参照 shadcn/ui 的组件编码风格：
- 使用 React.forwardRef 包裹
- 用 cva (class-variance-authority) 定义变体
- 用 cn() 工具函数合并 className
- 导出组件类型定义
- 文件结构参照 src/components/ui/card.tsx
```

### 2.3 更新 Skill 中的组件清单

组件创建完成后，别忘了更新 `DESIGN_SYSTEM.md` 中的可用组件列表。让 AI 重新扫描 `src/components/ui/` 目录，把新组件加进去。

```
请重新扫描 src/components/ui/ 目录，更新 DESIGN_SYSTEM.md 中的组件列表，
把新增的 StatsCard 组件加进去，包括它的 Props 说明和使用场景。
```

这样，下次 AI 需要展示统计数据时，就知道用 `StatsCard` 而不是从头写。

## 第三步：优化现有组件

项目用着用着，小明发现现有的 Button 组件缺少一些状态。他需要优化它，但又怕改坏。

### 优化流程

老师傅教了一套安全的优化流程：

**1. 先描述问题，不要直接改代码**

```
我们的 Button 组件（src/components/ui/button.tsx）需要优化：

当前问题：
- 没有 loading 变体，提交表单时无法显示加载状态
- 没有 icon-only 变体，工具栏的纯图标按钮样式不统一

期望效果：
- 新增 loading prop，为 true 时显示旋转图标并禁用点击
- 新增 size="icon" 变体，提供正方形的纯图标按钮样式

约束条件：
- 不能破坏现有的按钮用法
- 新增的变体要和现有变体兼容（如 loading + variant="destructive" 应该正常工作）
- 遵循项目的 INTERACTION_RULES.md 中的交互反馈规则
```

**2. 让 AI 先输出变更方案再动手**

```
请先分析当前 Button 组件的代码结构，输出你计划做的变更：
- 哪些代码需要修改
- 新增哪些代码
- 有哪些潜在的兼容性风险

不要直接修改代码，先让我确认方案。
```

**3. 验证优化结果**

优化完成后，让 AI 检查兼容性：

```
请检查项目中所有使用 Button 组件的地方，确认：
1. 现有用法是否都能正常工作
2. 新增的 loading 和 icon size 是否有使用示例
3. TypeScript 类型是否正确
```

## 设计与代码的交互验证

小明问了一个尖锐的问题："以前我们先在 Figma 里画设计稿，再让开发照着写代码。现在用 AI 直接写代码了，Figma 还有用吗？"

老师傅沉思了一下："Figma 当然还有用，但工作流变了。"

### 传统流程 vs AI 时代流程

**传统流程（瀑布式）：**
```
设计师画 Figma 稿 → 评审 → 开发按稿写代码 → 走查还原度 → 反复修改
```
问题：设计和代码是分离的，中间有大量翻译成本和信息损耗。

**AI 时代流程（交互式）：**
```
用自然语言描述需求 → AI 生成代码 → 浏览器里直接看效果 → 描述调整 → AI 修改 → 循环
```
设计和代码合二为一，所见即所得。

### 代码就是设计稿

老师傅说："在 AI 时代，代码本身就是最好的设计稿。它是可交互的，是真实的，不存在'还原度'的问题——因为你看到的就是真正会上线的东西。"

这并不意味着设计思考不重要了。恰恰相反，**设计思考变得更重要了**——因为你省掉了从 Figma 到代码的翻译时间，可以把更多精力花在思考"这样设计对不对"上。

### 实际的验证工作流

小明按照老师傅的建议，建立了这样的工作流：

**1. 描述 → 生成 → 验证**

```
用自然语言描述界面需求：
"创建一个用户个人资料页面，顶部显示头像和用户名，下面是个人信息表单，
包含姓名、邮箱、手机号。表单右下角有保存按钮。移动端单列布局。"

→ AI 生成代码
→ 在浏览器中查看效果
→ 发现问题："头像太小了，保存按钮不够明显"
→ 继续用自然语言调整
```

**2. 截图对比验证**

当你有参考设计时（不管是 Figma 稿、Dribbble 截图还是竞品截图），可以直接给 AI 看：

```
请参考这张截图的布局和风格，调整当前页面。注意：
- 保持我们项目的设计规范（颜色、间距、圆角按 DESIGN_SYSTEM.md）
- 只参考布局结构和信息层级，不要照搬颜色
```

**3. 跨设备验证**

```
请检查这个页面在以下尺寸下的表现：
- 移动端（375px）
- 平板（768px）
- 桌面端（1280px）

如果有布局问题，请直接修复。
```

### 设计工具的未来

小明问："那 Figma 这些设计工具是不是要被淘汰了？"

老师傅笑了："工具不会被淘汰，但使用方式会变。"

- **设计工具的新角色**：从"画最终稿"变成"快速探索创意"。当你需要同时比较 5 种布局方案时，Figma 的拖拽效率比写提示词高。
- **AI 设计工具的融合**：像 v0.dev 这样的工具，已经在弥合设计和代码的鸿沟——你描述需求，它直接给你可运行的代码。
- **核心转变**：以前是"设计先行，代码跟随"，现在是"设计和代码同步进行"。设计验证不再是看截图对比，而是直接在浏览器里操作真实界面。

老师傅总结说："**设计的本质是解决用户问题，不是画漂亮的图**。工具只是手段。AI 让你跳过了'画图→写代码'这个翻译步骤，但'理解用户→定义方案'这个思考步骤，永远不会被跳过。"

## 保持设计一致性的完整方案

把前面讲的串起来，老师傅帮小明总结了一套完整方案：

### 1. 建规则

制定视觉交互规则，写入 `INTERACTION_RULES.md`，涵盖颜色、间距、交互反馈、动效、组件使用五个维度。

### 2. 存记忆

把规则存入项目 Skill（5.6 节的 `my-project-design`），让 AI 在需要时自动加载。

### 3. 扩组件

项目需要的组件，先定义规格再让 AI 实现，遵循组件库的代码风格，完成后更新 Skill 中的组件清单。

### 4. 勤优化

发现组件不够用或有问题时，描述问题→确认方案→实施修改→验证兼容性。

### 5. 直接验证

用浏览器直接看效果，而不是对着设计稿数像素。代码就是设计的最终表达。

---

## 小结

小明学到了：

- **视觉交互规则**不只是颜色和间距，还包括交互反馈、动效和组件使用规范
- **新增组件**要先定义规格，遵循现有代码风格，完成后更新 Skill
- **优化组件**要描述问题而非直接改码，先确认方案再动手，最后验证兼容性
- **设计与代码可以直接交互验证**——在浏览器中看真实效果比对设计稿更高效
- **设计工具不会消失，但角色会变**——从画最终稿变成探索创意，核心设计思考永远不可替代

老师傅说："UI 最佳实践的核心很简单——**你定规则，AI 守规则，浏览器验规则**。规则越清晰，AI 的输出越一致。不要追求完美的设计稿，追求可运行的好体验。"

---

::: tip 实践建议
- 花 30 分钟制定你项目的视觉交互规则，写入 `INTERACTION_RULES.md`
- 新增组件时，先写"组件规格"提示词，包含 Props、交互规则、无障碍要求
- 优化组件时，让 AI 先输出变更方案，确认后再修改
- 养成在浏览器里直接验证的习惯，减少设计稿和代码之间的翻译成本
- 每次组件变化后，让 AI 重新扫描更新 Skill 中的文档
:::
